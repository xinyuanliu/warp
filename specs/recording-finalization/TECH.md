# Recording Finalization and Upload Reliability — Tech Spec
Product spec: `specs/recording-finalization/PRODUCT.md`

All code references are pinned to commit `b7430f40a9ef73a534f97bbc815944ebf17eedf8` in the `warp` repo (feature branch `varoon/va-robust-finalization`; identical commit to `varoon/va-blocklist-ui`, where the recording code was researched).

## Context
Computer-use video recording is wired up by the `start_recording`/`stop_recording` tool calls. The live capture is an ffmpeg `x11grab` sidecar whose handle lives in a runtime-global singleton, and finalization (stop → upload → report) is implemented only inside the stop executor. There is no teardown/cancel/limit hook, so any non-`stop_recording` exit orphans the capture. See `PRODUCT.md` for the user-visible behavior this must produce.

## Proposed changes

### 1. A `RecordingController` that owns the state machine (executor layer)
Promote `RecordingController` from a pure slot registry to the owner of the `Idle → Starting → Active → Finalizing → {Published|Discarded|Failed}` machine, but keep the async stop+upload work at the executor layer (`BlocklistAIActionExecutor`), which is the only place with `ServerApiProvider`/`ai_client`/conversation token. Concretely:
- Add a `FinalizeReason` enum: `StoppedByAgent`, `AgentFinished`, `LimitReached`, `FfmpegExited`, `Cancelled` (PRODUCT invariant 8). Each maps to a `termination_reason` string + completion status.
- Track a `Finalizing` marker in the controller so a second trigger for the same `recording_id` short-circuits (idempotency, invariant 3). `take_handle_or_err` already removes the handle atomically; add a `begin_finalize(recording_id) -> Option<Handle>` that returns the handle exactly once and records that finalization is underway (so `AgentFinished`/watcher/`Cancelled` racing `StoppedByAgent` cannot double-run).

### 2. Centralized, idempotent `finalize`
Factor the body of `stop_recording.rs (101-153)` into a shared async helper on the executor, e.g. `BlocklistAIActionExecutor::finalize_recording(recording_id, reason, conversation_id, ctx) -> RecordingTerminalOutcome`:
1. `begin_finalize` to claim the handle (no-op terminal if already finalized/absent).
2. `recorder.stop(handle)` for a graceful SIGINT finalize; on `Cancelled`/`FfmpegExited` where graceful stop isn't possible, fall back to reading whatever file exists.
3. **burn-in hook** (no-op today): a single call site where feature #3 will composite overlays onto the file before upload.
4. Upload via `FileArtifactUploader` (unchanged association logic).
5. **Unconditional temp-file cleanup** of the `.mp4` and its `.log` on every terminal branch (invariant 19) — today only the `.mp4` is removed and only on some paths.
6. Return a terminal outcome (`Published{artifact,...} | Discarded{reason} | Failed{error}`), carrying `termination_reason` derived from `reason`.

`stop_recording.rs` becomes a thin caller of `finalize(StoppedByAgent)`; the `ConversationNotSynced` early-return stays as-is (PRODUCT invariant 17). It is a defensive guard that is effectively unreachable in the cloud computer-use flow: the recording tool calls are server-emitted, and the server only emits them while running a turn for the conversation, which means the server conversation token has already been assigned (`history_model.rs:790-852` `set_server_conversation_token_for_conversation*`; token read at `controller.rs:2328-2329`). **Recommendation: keep** it as cheap insurance against a None-token unwrap rather than building retry/persistence around it; flagged for the user as keep-vs-drop.

### 3. Invocation points (wire finalize into every exit)
- **StopRecording** (existing): calls `finalize(StoppedByAgent)`.
- **Cancel funnels**: `cancel_all_running_async_actions_for_conversation` (`execute.rs:919`) must, after handling the async map, also finalize an `Active` recording that is *not* in `async_executing_actions` (the current gap). Add: if the controller has an `Active`/`Starting` recording for this conversation's runtime, call `finalize(Cancelled)` (or release+cleanup for `Starting`, invariant 14). This covers `send_query` follow-up, `cancel_conversation_progress`, and `fail_conversation_due_to_shell_exit`, which all route through `cancel_all_pending_actions` → this method.
- **ffmpeg-exit watcher** (invariants 11–12): see §5.
- **Driver run tail** (invariant 10, 21): at `driver.rs:895`, alongside `run_snapshot_upload`, add a `finalize_active_recordings()` step that finalizes any still-`Active` recording as `AgentFinished` and then awaits the artifact-upload registry (§4), time-boxed like `snapshot_upload_timeout`. This runs before `tx.send(result)` so uploads are not abandoned by process termination.

### 4. Generalized in-flight upload registry (`ArtifactUploadState`)
The stop path spawns its upload as a detached background future (per `ctx.spawn` semantics: the sole `AbortHandle` is dropped at `execute.rs:812`, so cancellation never aborts it — the upload runs to completion but its result can be discarded, and it dies with the process on teardown). To make teardown wait on it:
- Introduce a runtime singleton `ArtifactUploadState` that tracks in-flight uploads (a counter + a `Notify`/`Barrier`, or a set of join signals). Every artifact upload — recording finalize, `report_screenshot`, `upload_artifact` — registers on start and deregisters on completion. This generalizes the drain to all artifact types (invariant 21), not just video.
- The driver run tail (`driver.rs:895`) awaits `ArtifactUploadState::drain(timeout)` after `finalize_active_recordings()`; on timeout it proceeds and the affected outcome is `Failed` (invariant 22).

### 5. ffmpeg-exit watcher (modeled on the LRC block watcher)
Mirror `ShellCommandExecutor::action_result_future` (`shell_command.rs:515-634`), which `select!`s a completion oneshot against a timeout:
- Have the Linux recorder, when building the handle (`recording.rs:102-108`), also spawn a monitor that `await`s `child.wait()` and fires a `oneshot`/`watch` stored on `RecordingHandle` (the `Child` isn't `Clone`, so the signal must be produced by the recorder that owns it; noop/mac recorders never fire). Expose it as e.g. `RecordingHandle::exited()`.
- After `finish_start` (`start_recording.rs:84-86`), the executor spawns a task that awaits that signal and calls `finalize(LimitReached | FfmpegExited)` — distinguishing the two by whether ffmpeg's exit was clean at the `-t`/`-fs` cap vs a nonzero/killed exit. This replaces today's lazy detection (only at the next `stop`, `recording.rs:123-126`).

### 6. Terminal result emitted off the action map
Today the async completion callback early-returns if the action was removed from `async_executing_actions` (`execute.rs:813-815`), discarding the result on cancel. Change finalize so the terminal `RecordingTerminalOutcome` is surfaced through the controller/telemetry path regardless of whether a tracked tool call still exists (PRODUCT invariant 16). When a `stop_recording` tool call is live, its result is the outcome; when finalize is triggered without one (`AgentFinished`/watcher/`Cancelled`), publish the artifact (association with the conversation makes it viewable) and emit a synthesized terminal outcome for telemetry/logging.

**Agent notification vs the LRC precedent (PRODUCT invariant 11).** LRC surfaces async completion by resolving a *live* tool call: `RequestCommandOutput`'s async future resolves later and emits `FinishedAction` with the *original* action id (`execute.rs:812-826`); that result is stored (`action_model.rs:1278-1287`), drained on the next request (`drain_finished_action_results`), and correlated by the server to the `run_command` tool call the agent issued, so the agent's next turn sees it. The recording watcher/`LimitReached` case has **no live tool call** to resolve (`start_recording` already returned the `recording_id`; no `stop_recording` is pending), so this mechanism does not transfer directly. Two options:
- (a) **Default, no new wiring:** the client publishes the artifact immediately and the agent finds out the next time it references the recording — a later `stop_recording`/`finish_computer_use` returns a benign `StoppedEarly`/no-op result. This is what the state machine already yields.
- (b) **Proactive push (optional):** to notify the agent mid-sequence (before it next stops/finishes), a notice must be injected into the agent's next request without a pending tool call. Prior art for exactly this channel exists — `OrchestrationEventService` buffers per-conversation events and drains them into the next request as `AIAgentInput::MessagesReceivedFromAgents`/`EventsFromAgents` (`orchestration_events.rs:281-323`), first-class inputs (not tool results) with server-echo delivery confirmation + retry (`orchestration_events.rs:328-427`); this is the same path that delivers inter-agent inbox messages. It is not a drop-in, though: it is scoped to conversation-level orchestration agent-runs and flushes on a client-initiated request, whereas the computer-use subagent's turns are server-driven off tool-call results — so wiring a recording notice to the subagent needs either a new event/input kind plumbed to the subagent or a server-synthesized message. So (b) is "proven channel + net-new wiring," not free. Recommend (a) as the baseline; do (b) only if product wants mid-sequence notification (PRODUCT invariant 11 open question).

### 7. Cancellation UX / agent guidance
On `finalize(Cancelled)`, set a `termination_reason` that says the recording was ended by interruption (invariant 24). Add a one-line instruction to the computer-use agent prompt (`warp-server` `computer_use_agent.yaml`) telling the agent that an interrupted recording is not resumable and it should call `start_recording` again if it still needs a video. This is a cross-repo change tracked with this feature.

## Parallelization
Not beneficial. The change is one tightly-coupled vertical slice — the controller state machine, the factored-out `finalize`, its invocation points, and the shared upload registry all touch the same executor/controller/recorder types and must land together to preserve invariants (a half-wired finalize is worse than today). The only separable pieces are the cross-repo `warp-server` prompt tweak (§7) and the segmented-upload follow-up, neither of which is worth a parallel agent. Recommend a single agent on the `varoon/va-robust-finalization` branch (this worktree), one PR for the client change plus a small companion PR for the server prompt line.

## Risks and mitigations
- **Hard sandbox TTL kill mid-capture** (PRODUCT invariant 25) — genuinely distinct from graceful teardown, and a drain is provably unreachable on this path. On agent finish / idle-timeout the driver run tail (`driver.rs:895`) gracefully stops capture and drains uploads before `tx.send`, so nothing is lost. But `MaxInstanceRuntime` is passed as `CreateParams.Timeout` to the docker-sandbox service (`dockersandbox.go:298-308` → `client.Create` at `tracing.go:68-87`), and the SDK (a warpdotdev fork of `docker/go-sdk/sandboxes`) documents `Timeout` as "how long the sandbox should live before being automatically deleted" with a service-owned `ExpiresAt` (`client.go:392-394`, `sandbox.go:40-41`). So the TTL is enforced **by the sandbox service, not by any warp-server Go code**: the container is auto-deleted at `ExpiresAt`, and the server only *notices afterward* via the poller finding it `NotFound` → `SandboxExpiredError` → `finalizeExecutionCompletion` (`poller.go:91-112`). No warp-server teardown (`cleanupSandbox → deleteSandbox`, `dockersandbox.go:671-699`) runs before the container dies on this path, and the SDK exposes no stop-grace on sandbox deletion (its only SIGTERM/SIGKILL semantics are for explicit *process* `Terminate`/`Kill`, not sandbox auto-delete); the API-key `+5m` buffer (`task_utils.go:38-41,162`) is key lifetime only, not container grace. Therefore an `Active` recording at hard-kill time is a **total loss no drain can recover**: capture was never SIGINT-finalized so the file is truncated/unplayable (moov atom is written only on graceful SIGINT, `recording.rs:4-5`), and nothing was uploaded (a single upload happens only at finalization). Only the segmented/streaming-upload follow-up closes this window. Cheap partial mitigation without streaming: keep `MaxInstanceRuntime` comfortably above the recording duration cap (`-t`, default 10m) so an active recording rarely coincides with the TTL; the `-t`/`-fs` caps also bound resource use.
- **Idempotency races**: `AgentFinished`/watcher/`Cancelled`/`StoppedByAgent` can fire concurrently. Mitigated by claiming the handle exactly once via `begin_finalize` (§1); losers observe "already finalizing" and no-op.
- **Drain wedging teardown**: bounded by the upload-time budget (§4); on timeout the run proceeds and marks `Failed`.
- **Graceful-stop latency at teardown**: `recorder.stop` waits for ffmpeg to flush the moov atom (SIGINT + `STOP_TIMEOUT`). The run-tail finalize inherits that; keep it inside the drain budget.

## Follow-ups
**Build order: drain now, streaming later.** Build the centralized finalize + run-tail drain (§2–§4) first: it is the only thing that fixes the common cases — graceful agent finish, idle-timeout, finish-without-stop, and user/new-query/shell-exit cancel — plus temp-file cleanup and the never-terminal orphan, none of which streaming obviates. It is not throwaway: even with streaming you still need the drain to finalize gracefully and flush the final segment. Defer streaming as a scoped fast-follow that only closes the rare hard-kill-mid-active window.
- **Segmented/streaming upload during capture** — the only defense against hard-TTL/sudden-death mid-recording; bounds loss to the last un-flushed segment. The existing `FileArtifactUploader` flow does **not** reuse cleanly for a live stream: it is a finished-file contract (create target → PUT the whole on-disk file → `confirm_file_artifact_upload(artifact_uid, checksum)` over the completed blob, `artifact_upload.rs:105-182`), which needs a complete file and a final checksum and supports whole-file retry precisely because the file is buffered on disk. A live single fMP4 PUT loses that retry (a dropped connection can't resend from a buffer you no longer hold) and needs new server support (resumable/multipart upload + finalizing a growing object into a file-artifact record + signed URL). A **hybrid** (disk-buffered periodic segments) preserves per-segment retry and can reuse `upload_file_to_target` per segment, but produces N objects, so playback needs either multiple artifacts or server-side concatenation/HLS-style manifesting — less risky than live streaming but still net-new server + playback work. ffmpeg also currently muxes a single MP4 finalized only on SIGINT (`recording.rs`), so either approach needs segment muxing or periodic remux.
- **Retain-and-retry local file on upload failure** — resolves the existing `TODO(vkodithala)` at `stop_recording.rs:124`; would upgrade some `Failed` outcomes to eventual `Published` via the registry.
- **Burn-in overlay (#3)** hooks the finalize burn-in call site (§2 step 3).
- **Metrics** for finalization causes/outcomes (separate change, out of scope here).
